# Design: Prefer Native ZAX D8 Maps

**Issue:** #80 — Prefer native ZAX D8 maps for source mapping and stepping  
**Depends on:** #78 (PR #82, merged), #79 (PR #85, merged), #81 (PR #83, merged)  
**Upstream:** ZAX#1349 (segment.line population — already present in current ZAX output)  
**Type:** Behaviour refinement (improve ZAX D8 map path, keep asm80 path unchanged)

---

## Motivation

ZAX generates a native `.d8.json` alongside its `.hex` and `.lst` artifacts. Debug80
already knows how to load and consume D8 debug maps — `loadDebugMap()` reads them,
`buildMappingFromD8DebugMap()` converts them to internal segment/anchor structures, and
`buildSourceMapIndex()` indexes them for breakpoint resolution.

The current flow **already works for ZAX builds** when both artifacts exist. However,
the mapping pipeline was designed around the asm80 workflow where the `.lst` is the
primary mapping source and the `.d8.json` is a debug80-generated cache of parsed LST data.
For ZAX, the relationship is inverted: the `.d8.json` is the native, authoritative mapping
source, and the `.lst` is a secondary human-readable artifact.

This issue addresses three concerns:

1. **Staleness semantics** — for ZAX, the `.d8.json` is not derived from the `.lst`, so
   "D8 map is older than LST → regenerate from LST" is wrong. A ZAX `.d8.json` should
   never be regenerated from the `.lst`.

2. **Consumer-side `lstLine` → `line` fallback** — per the issue comment, if a ZAX D8 map
   has `lstLine` populated but `line` is missing for some segments, debug80 should fall
   back to using `lstLine` as the source line. (Current ZAX output does populate `line`,
   but a defensive consumer-side fallback protects against older ZAX versions.)

3. **Skip unnecessary fallback logic** — when a valid native `.d8.json` exists, the
   asm80-specific Layer 2 matching, source fallback, and in-process `compileMappingInProcess`
   paths should not run. This is already the case in the happy path, but making it explicit
   by detecting a "native" (externally-generated) D8 map makes the flow clearer.

## Current Flow Analysis

### What already works

When ZAX produces `main.hex`, `main.lst`, and `main.d8.json` simultaneously:

1. Adapter checks `fs.existsSync(hexPath) && fs.existsSync(listingPath)` — both exist. ✓
2. `loadProgramArtifacts` reads `.lst` → `ListingInfo` (lineToAddress map). ✓
3. `buildMappingFromListing` calls `service.resolveDebugMapPath(...)` → path to `.d8.json`.
4. `isDebugMapStale(mapPath, listingPath)` — both have identical mtimes → returns `false`. ✓
5. `loadDebugMap(mapPath)` — finds and parses the ZAX `.d8.json`. ✓
6. `buildMappingFromD8DebugMap()` — builds segments with `loc.line` from `segment.line`. ✓
7. `buildSourceMapIndex()` — indexes on `loc.file` + `loc.line`. ✓
8. Breakpoint resolution via `resolveLocation()` — works if `loc.line` is populated. ✓

### What could go wrong

| Scenario | Risk |
|----------|------|
| ZAX `.d8.json` is deleted but `.lst` remains | Regeneration from `.lst` produces poor mapping (no Layer 2 anchors, no source paths). The `.lst` is ZAX-format, not asm80-format — the listing parser may produce garbage. |
| User edits `.zax` and re-assembles, `.lst` gets newer mtime than `.d8.json` by a file-system tick | `isDebugMapStale` returns `true`, discards the native `.d8.json`, regenerates from `.lst` — wrong. |
| ZAX D8 map has `lstLine` but `line` is null for some segments | Those segments are skipped by `buildSourceMapIndex` → breakpoints on those lines silently fail. |
| ZAX `.lst` format differs from asm80 `.lst` | `parseListing()` may produce incorrect `ListingInfo`. The listing-line breakpoint path (direct `.lst` file debugging) may be unreliable. |

## Design

### 1. Detect native (externally-generated) D8 maps

Add a predicate to identify D8 maps that were NOT generated by debug80:

```typescript
function isNativeDebugMap(map: D8DebugMap): boolean {
  const gen = map.generator;
  if (gen === undefined) return true; // no generator → assume native
  const name = (gen as Record<string, unknown>).name;
  if (typeof name === 'string' && name === 'debug80') return false;
  return true;
}
```

ZAX D8 maps have `generator.tool: 'zax'` (no `name` field or `name !== 'debug80'`).
Debug80-generated maps have `generator.name: 'debug80'`.

### 2. Skip staleness regeneration for native D8 maps

In `buildMappingFromListing()`, change the flow so that when a valid D8 map is loaded
and it is **native** (not generated by debug80), the staleness check does not lead to
regeneration. The map is used as-is:

```
mapPath = resolveDebugMapPath(...)
if (fs.existsSync(mapPath)):
    map = loadDebugMap(mapPath)
    if map is valid AND isNativeDebugMap(map):
        // Use directly — never regenerate from LST
        // Log: "Using native D8 map from <generator>"
    else:
        // Existing flow: check staleness vs LST, regenerate if stale
else:
    // Existing flow: build from LST
```

This means:
- ZAX `.d8.json` → loaded and used directly, never overwritten.
- debug80-generated `.d8.json` → existing staleness/regeneration logic applies.

### 3. Consumer-side `lstLine` → `line` fallback

In `buildMappingFromGroupedDebugMap()` in `d8-map.ts`, add a defensive fallback:

```typescript
const line = segment.line ?? segment.lstLine ?? null;
```

Currently the code reads:
```typescript
const line = segment.line ?? null;
```

This single change means: if `segment.line` is missing/null, fall back to `lstLine`
as the source line. For debug80-generated maps, `line` is always populated (from
Layer 2 matching), so this is a no-op. For ZAX maps where `line` happens to be missing,
this provides a usable fallback.

### 4. Don't overwrite native D8 maps

Currently, when the D8 map fails to load or is stale, `buildMappingFromListing()` calls
`writeDebugMap()` to write a fresh debug80-generated map to the same path. For ZAX builds,
this would destructively overwrite the native `.d8.json` with an inferior debug80-generated
one.

Add a guard: do not write a debug80-generated map if the path already contains a native D8
map. Alternatively, track whether the map was loaded natively and skip the write path.

The simplest approach: add a `skipWrite` flag to the flow:

```
if (!debugMap):
    // build from LST as before
    writeDebugMap(...)  // write debug80-generated map
elif isNativeDebugMap(debugMap):
    // loaded a native map — do NOT write over it
else:
    // loaded a debug80-generated map — existing flow
```

### 5. Log native map usage

When a native D8 map is loaded, log it for diagnostics:

```
Debug80: Using native D8 map from "zax" at "<mapPath>".
```

## Test Requirements

### New tests (`tests/debug/native-d8-map.test.ts`)

1. **Native map detection:** `isNativeDebugMap()` returns `true` for a map with
   `generator: { tool: 'zax' }` and `true` for a map with no `generator` field.
2. **Native map detection (debug80):** Returns `false` for map with
   `generator: { name: 'debug80' }`.
3. **Staleness skip:** When a native d8.json exists and is older than the listing,
   `buildMappingFromListing()` still uses the native map (does not regenerate).
4. **No overwrite:** When a native d8.json exists, `writeDebugMap()` is NOT called.
5. **Regeneration for debug80 maps:** When a debug80-generated d8.json is older than
   the listing, the existing regeneration path still runs.

### Updated tests (`tests/mapping/d8-map.test.ts`)

6. **lstLine fallback:** A segment with `line: undefined` and `lstLine: 42` produces
   a `SourceMapSegment` with `loc.line === 42`.
7. **line takes precedence:** A segment with `line: 10` and `lstLine: 42` produces
   a `SourceMapSegment` with `loc.line === 10`.
8. **Both null:** A segment with `line: undefined` and `lstLine: undefined` produces
   `loc.line === null`.

### Existing tests

9. All existing mapping-service, d8-map, breakpoint-manager, and program-loader tests
   must pass unchanged.

## Acceptance Criteria (from issue #80)

- [ ] Breakpoints can be set in `.zax` source and resolve to executable addresses.
- [ ] Stepping lands on `.zax` source lines using the native D8 map.
- [ ] debug80 does not require asm80-specific mapping fallback for working ZAX artifacts.
- [ ] Tests cover D8 map loading and source line resolution for ZAX outputs.

---

## What This Does NOT Do

- Does not make the `.lst` file optional for launching (the adapter still requires it).
  Making listing optional is a larger change that can be explored separately.
- Does not change asm80 behaviour in any way.
- Does not add ZAX in-process compilation.
- Does not modify the D8 map format specification.
