# AZMDoc metadata comments

Status: draft standard
Date: 2026-05-21

## Purpose

AZMDoc is the small machine-readable comment format used by AZM register-care
tools. Human prose remains ordinary semicolon comments. Register contracts use a
separate compact `;!` form so the analyzer never has to mine prose for meaning.

AZMDoc metadata must not change generated bytes. It only describes the intended
register contract at routine boundaries and selected call sites.

## Source Contracts

Routine source contracts are contiguous `;!` lines immediately before a routine
entry label:

```asm
; Tests candidate active-piece placement against walls, floor, and board rows.
; D contains the candidate x coordinate.
; E contains the candidate y coordinate.
; Carry is returned set when placement is blocked.
;!      in        DE
;!      out       carry
;!      clobbers  A
@CHECK_COLLISION_AT_DE:
```

The supported source contract keys are:

```text
in
out
clobbers
preserves
```

The old `@in`, `@out`, `@clobbers`, `@preserves`, `@proc`, `@extern`, and
divider-block spellings are not part of the AZM source contract language.

## Routine Entry Labels

The preferred AZM routine-entry spelling is `@Name:`. The callable symbol is
still `Name`; the `@` prefix marks the label as an explicit routine entry for
AZM tools. Plain labels remain valid ASM80-style labels, but source annotation
tools should be more conservative about inserting generated contracts above
plain labels unless a human prose block is already present.

Plain internal labels are branch targets inside the current routine. They should
not receive generated AZM contract blocks. If a branch target needs an
independent contract, promote it into an explicit routine entry.

## External Interfaces

External register-care contracts live in `.asmi` files. `.asmi` is not assembly
source and does not use comment leaders:

```text
extern MON_PRINT_CHAR
in A
clobbers A
end

extern MON_GET_KEY
out A
out zero
clobbers carry
end
```

Use `.asmi` for MON3 or library calls whose body is unavailable to the analyzer.
Do not put external contracts in `.asm` comments. `.asmi` is deliberately
comment-free; every non-blank line must be part of an `extern` contract record.

## Caller Hints

Caller-local output hints use the narrow `expects out` comment immediately
before the relevant call:

```asm
    ; expects out DE
    call NORMALISE_COORD
```

This is a suppression for an ambiguous call site, not the preferred way to
document a stable API. If many call sites need the same hint, add a source
contract to the callee or a `.asmi` extern contract.

## Carrier Lists

Carrier lists use compact comma-separated spelling:

```asm
;!      in        A,DE,HL
;!      out       carry
;!      clobbers  BC
```

Register pairs are shorthand for their constituent carriers:

```text
BC = B,C
DE = D,E
HL = H,L
IX = IXH,IXL
IY = IYH,IYL
SP = SPH,SPL
```

Flags may be named individually:

```text
carry zero sign parity halfCarry
```

Use `carry` for the carry flag and `C` for register C. Generated metadata should
avoid `F` and `AF`; when a flag is meaningful, name the individual flag.

## Contract Meaning

`in` declares registers or flags whose incoming value is meaningful to the
routine.

`out` declares registers or flags that carry meaningful returned values. An
output is not an accidental clobber. If a routine intentionally transforms a
register in place, the same carrier may appear in both `in` and `out`.

`clobbers` declares externally visible scratch damage. Callers must preserve
these carriers themselves if they need the incoming values after the call.

`preserves` declares carriers restored to their incoming value before return.
Generated source-facing contracts should normally omit `preserves`; unlisted
carriers are assumed not to be part of the routine's visible damage.

## Tooling Behavior

AZMDoc-aware tools should:

- parse only compact `;!` source contract lines in `.asm` and `.z80` files
- parse bare `extern` records in `.asmi` files
- treat contiguous generated `;!` source contract lines as tool-owned and
  replaceable
- preserve human prose comments when emitting source
- never change generated bytes because of AZMDoc metadata alone
- prefer warnings before hard failures while a source tree is being annotated

AZMDoc gives tools structured register-care information, while the source
remains ordinary Z80 assembly with comments.
